# Solo Backlog

## Источник правды

По умолчанию: `docs/BACKLOG.md`. Альтернативы (Notion, Obsidian, GitHub Issues) — указать в README проекта.

## Формат задачи

```markdown
### [ID] Название задачи
- **Статус**: backlog | in_progress | blocked | done
- **Приоритет**: P0 | P1 | P2 | P3
- **Ветка**: feature/название (если in_progress)
- **Критерии готовности**:
  - [ ] ...
- **Заметки**: ...
```

## WIP-лимит

- Максимум **2** задачи в `in_progress`.
- Новая задача — только если WIP < 2 и текущая завершена или явно отложена.

## Workflow

### Старт сессии
1. Прочитай `docs/BACKLOG.md`
2. Покажи текущий WIP и следующую задачу по приоритету
3. Спроси, с чем работаем

### Добавление задачи
1. Присвой ID (следующий номер)
2. Определи приоритет
3. Запиши критерии готовности

### Завершение задачи
1. Проверь критерии готовности
2. Статус → `done`
3. Запись в `docs/SESSION-LOG.md`

### Синхронизация с GitHub Issues
Если backlog в Issues:
```bash
gh issue list --state open
gh issue create --title "..." --body "..."
```

## Приоритизация

1. **P0** — блокер, прод не работает
2. **P1** — текущий спринт / активная фича
3. **P2** — важно, но не срочно
4. **P3** — идеи, техдолг, nice-to-have
