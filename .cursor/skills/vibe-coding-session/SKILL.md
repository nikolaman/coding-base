# Сессия вайб-кодинга

## Золотое правило

AI пишет быстрее, чем думает. Ты должен думать быстрее, чем принимаешь его код.

## Цикл (строго по порядку)

```
Сформулировал задачу → AI предложил план → Проверил план → AI написал код → Тесты + review → Коммит
```

## Шаг 1: Требования

Перед кодом зафиксируй:
- Что делаем и зачем
- Критерии готовности (acceptance criteria)
- Ограничения (стек, производительность, безопасность)
- Что **не** входит в scope (YAGNI)

Обнови `docs/BACKLOG.md` — статус задачи `in_progress`.

## Шаг 2: План

Выдай план **без кода**:
- Какие файлы затронуты (≤5 за шаг)
- Оценка объёма (цель: 100–300 строк)
- Риски и edge cases
- Нужны ли тесты / ADR

Дождись подтверждения плана.

## Шаг 3: Реализация

- Одно логическое изменение за шаг
- Следуй существующим конвенциям проекта
- Не добавляй зависимости без обоснования

## Шаг 4: Проверка

Чеклист:
- [ ] Тесты запущены и проходят
- [ ] Код понятен (объясни неочевидное)
- [ ] Edge cases проверены
- [ ] Новые зависимости проверены
- [ ] Diff ≤300 строк (иначе — разбить)

При крупном diff — запусти Bugbot review.

## Шаг 5: Фиксация

1. Запись в `docs/SESSION-LOG.md`:
   - Дата, задача, решения, открытые вопросы
2. ADR — если архитектурное решение (скилл `adr-documentation`)
3. Обновить `docs/BACKLOG.md` — статус `done` или `blocked`
4. Коммит — только по запросу пользователя

## Антипаттерны

- ❌ Сразу генерировать 10 файлов
- ❌ Коммитить без тестов
- ❌ Принимать код без понимания
- ❌ Смешивать рефакторинг с фичей
- ❌ Начинать новую задачу без закрытия текущей
