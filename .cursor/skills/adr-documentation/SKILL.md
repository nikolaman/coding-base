# ADR — Architecture Decision Records

## Когда писать ADR

- Выбор технологии / библиотеки / паттерна
- Изменение архитектуры (модули, API, БД)
- Компромисс с долгосрочными последствиями
- Отказ от альтернативы (почему не X)

Не нужен ADR для: мелких багфиксов, переименований, очевидных решений.

## Процесс

1. Скопируй `docs/adr/0000-template.md` → `docs/adr/NNNN-краткое-название.md`
2. NNNN — следующий порядковый номер (4 цифры)
3. Заполни все секции
4. Статус: `proposed` → `accepted` (после ревью) или `deprecated` / `superseded`

## Шаблон секций

- **Контекст** — проблема и ограничения
- **Решение** — что выбрали
- **Альтернативы** — что рассматривали и почему отвергли
- **Последствия** — плюсы, минусы, риски
- **Связанные решения** — ссылки на другие ADR

## После принятия

- Ссылка в README или `docs/architecture/README.md`
- Запись в `docs/SESSION-LOG.md`

## Индекс

Поддерживай список в `docs/adr/README.md`:

```markdown
| ADR | Статус | Название |
|-----|--------|----------|
| 0001 | accepted | Выбор ORM |
```
